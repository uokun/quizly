# 要件定義書: Quizly

## プロジェクトの目的
ユーザーが教材画像からクイズデータ（JSON形式）をAIを用いて自動生成し、ブラウザ上でプレイ・編集・共有できる教育用Webアプリケーション「Quizly」の設計を行う。
本プロジェクトでは**実装フェーズを含まず、要件定義および技術の策定（詳細設計の入り口）まで**をスコープとする。

## ターゲット技術スタック変更
フレームワークや高度な状態管理ライブラリは使用せず、依存を最小限に抑える構成とする。

| 領域 | 構成 | 採用理由・方針 |
| :--- | :--- | :--- |
| **フロントエンド** | Vanilla HTML / CSS / JavaScript | フレームワークの学習コストや依存を排し、シンプルなWeb標準技術で構築する。 |
| **バックエンド/DB** | Firebase (Firestore, Functions, Auth, Hosting) | サーバーレスでJSONデータの保存・呼び出し、ユーザー管理を容易に行う。 |
| **AI連携** | Gemini API (自動生成, チューター) / Nano Banana (画像生成) | JSON生成と教育用画像（グラフ等）のプロンプト出力を連携させるため。 |

## アプリケーション要件詳細

### 1. 全体UI/UX方針
- **スマホライクなレイアウト**: モバイル端末での閲覧・操作を最優先に設計（スワイプ操作などを想定）。
- **ダークモード/ライトモード対応**: ユーザーの好みに応じた表示切り替え。

### 2. 画面フローと画面一覧詳細

#### 2.1. メイン（トップ）画面 / 未ログイン状態
- **目的**: LP（ランディングページ）として機能し、サービス価値を直感的に伝える。
- **UIイメージ**: 流行のAIサービス（ClaudeやChatGPT等）のような、洗練された単一カラムレイアウト。
- **機能・アクション**: 
  - 画面中央にプロンプトボックス風のアイキャッチを配置。
  - 「Googleでログイン (Sign in with Google)」ボタンを配置（Firebase Auth連携）。
  - すでにログイン状態であれば、自動的にダッシュボードへリダイレクト。

#### 2.2. ダッシュボード画面（ログイン後ホーム）
- **目的**: ユーザーのハブとなる画面。自身のクイズ管理と他者のクイズ探索を行う。
- **機能/表示内容**:
  - **新規作成への導線**: 画面下部または目立つ位置に「チャットでクイズを作成」ボタン・入力欄を配置。
  - **パーソナル領域**: 過去に自身で作成・保存したクイズ、および自由に作成したフォルダ群のツリー表示。
  - **共有領域**: 所属組織のクイズ、またはパブリックに公開されているクイズのタイムライン/検索機能。
  - **通知・ランキング（オプション）**: いいね/フォロー等の通知や、組織内のスコアランキング表示。

#### 2.3. クイズ新規作成画面（AIチャットUI）
- **目的**: 画像やテキストから対話形式でクイズ（JSON）を生成する。
- **UI/UX**: 画面下部に固定された汎用的なAIチャットボックス（入力フィールド＋送信/画像添付ボタン）。
- **フロー詳細**:
  1. ユーザーが入力欄のクリップアイコン等から教材画像（複数可）をアップロード。またはテキストで「江戸時代の基礎」等の意図を入力。
  2. チャット上で指示を追加（例:「難易度を上げて」「選択問題のみにして」）。
  3. AIが分析中（ローディングスピナーやプレースホルダー）の表現を行い、結果（JSON定義ベースの問題群）をチャット上にカード形式またはモーダル等でインライン表示。
  4. ユーザーはローカルでAIとやり取りを重ね、内容に納得したら「保存して完了」アクションを実行。この時点で初めてFirebaseの `quizzes/` コレクションに永続化される。

#### 2.4. クイズプレビュー・編集画面 (エディタ)
- **目的**: AIが自動生成したクイズデータ、あるいは既存のクイズを手動で詳細調整する。
- **機能**:
  - タイトル、公開範囲（Private/Org/Public）、所属フォルダの変更。
  - 問題文、選択肢、正解設定、AI解説文テキストの直接編集。
  - 問題形式（選択、穴埋め、並べ替え、記述、○×、線つなぎ）ごとの専用入力フィールド切り替え。
  - 難易度やタグの追加・削除。
  - Nano Banana等の画像・グラフ生成用プロンプト（`imagePrompt`）の確認と手動編集。

#### 2.5. クイズプレイ画面 (モバイル特化)
- **目的**: 実際にクイズを解き、学習効果を最大化する。
- **UX**: 1問1画面ベース。フリック/スワイプでのなめらかなページ遷移。
- **機能**:
  - 解答の入力/選択アクション。即時フィードバック有無の設定。
  - **AIチューターチャット**: 画面端のフローティングボタン等から呼び出し、間違えた問題に対して「なぜ違うのか」をAIにその場で質問できる機能。
  - **TTS（音声読み上げ）**: 速度調整（1.0x, 1.2x, 1.5x 等）などのコントロールUI。問題文や英単語の読み上げ。

#### 2.6. リザルト（結果）画面
- **目的**: 成績のフィードバックとモチベーション向上、復習の促進。
- **機能/表示内容**:
  - 正解数 / スコア / ランク表示。
  - エラー分析（間違えた問題へのショートカット復習リンク）。
  - 再挑戦やクイズ共有（SNS連携、ユニークリンクのコピー）機能。

### 3. データ構造設計 (JSON Schema & データベース構造)

Firebase Firestore上で自由度の高い階層管理を実現しつつ、AIが単一のプロンプト出力として破綻せずに生成できるデータモデルを定義します。

#### 3.1. Firestore コレクション構造（フォルダ・ファイルシステム管理）
固定の階層ではなく、OSのファイルシステムのようなネスト可能なフォルダ構造と権限管理を実現します。

*   `users/` (ユーザーメタデータ): `{ id, displayName, currentOrganizationId }`
*   `organizations/` (組織): `{ id, name, type (school/company), memberIds }`
*   `folders/` (自由作成可能な階層フォルダ): 
    - `{ id, name, ownerId (UserId または OrgId), parentId (ルートの場合は null), createdAt }`
    - これにより「自分のフォルダの中に別のフォルダ」を作ることが可能。
*   `quizzes/` (クイズメタデータ・本体): 
    - `{ id, title, folderId (配置先フォルダ), authorId, visibility (private/org/public), data (以下のAI出力JSONを文字列化またはMapとして保持), createdAt, updatedAt }`

#### 3.2. クイズデータ本体 (AI出力 JSON Schema 詳細版)
クイズの中身（問題のリスト）は1つのJSONドキュメント（またはMap）として保持します。AIには以下の構造に合致するよう生成させます。

```json
{
  "title": "（AIが判定したクイズのタイトル）",
  "difficultyLevel": "基礎", // 基礎 | 応用 | 入試レベル等
  "subjectArea": "日本史", // 教科やカテゴリのタグ
  "questions": [
    {
      "id": "q_123456",
      "type": "multiple_choice", // multiple_choice, fill_in, ordering, free_response, true_false, matching
      "questionText": "江戸幕府を開いた人物は誰ですか？",
      "options": [
        { "id": "opt_A", "text": "豊臣秀吉" },
        { "id": "opt_B", "text": "徳川家康" },
        { "id": "opt_C", "text": "織田信長" }
      ],
      "correctAnswers": ["opt_B"], // 複数正解を考慮し配列とする。free_responseの場合は模範解答の文字列配列
      "explanationText": "関ヶ原の戦いで勝利した徳川家康が1603年に江戸に幕府を開きました。", // 正解の解説。チューターの初期コンテキスト
      "imagePrompt": null // 画像やグラフが不要な場合は null。必要な場合は "A minimalist map of Japan focusing on Edo, educational diagram" のような英語プロンプト。
    }
  ]
}
```

### 4. AIエンジンの制御とプロンプト方針

Geminiに対して、安定して上記の複雑なJSONを出力させ、かつ教育的に正しい振る舞いをさせるためのプロンプト設計方針です。

#### 4.1. 自動生成プロンプト（教材画像入力 → JSON出力）
- **役割付与**: 「あなたは優秀な教育コンテンツクリエイターです。提供された資料から、学習者のレベルに合わせた最適なクイズを作成してください。」
- **出力制約 (System Instruction)**:
  - 返却フォーマットは必須で規定のJSONスキーマのみとする。マークダウンの ```json ``` 等が付与された場合もフロントエンド側でクレンジング処理を入れる。
  - OCR情報から単純な単語・意味の変換だけでなく、資料の「意図」を汲み取った問題（例：図表があれば、その数値を読解させる選択問題）を生成させる。
- **画像問題（Nano Banana）対応ルール**:
  - `[図解が必要な事象]`, `[グラフ化できるデータ]` を検知した場合、必須で各要素の `imagePrompt` フィールドに画像生成AI（例: Nano Banana）でそのまま使える英語のプロンプトを記述させる。

#### 4.2. AIチュータープロンプト（プレイ中の質問・解説）
- **コンテキストの注入**: チューター呼び出し時、裏側のシステムプロンプトに「現在の問題（`questionText`等）」「ユーザーの選んだ解答」「正解(`correctAnswers`)」「解説(`explanationText`)」をJSON形式のまま不可視のコンテキストとして付与してAPIを叩く。
- **教育的振る舞いの強制**: 
  - 単に答えをポンと教えるのではなく、ソクラテス的問答法（ヒントを与え、ユーザーに気づかせる）を促すシステムプロンプトを設定する。

### 5. Vanilla JS環境下でのUI/UX実装要件 (Mobile First)

フレームワークを用いないVanilla環境でのモバイル最適化の重要項目です。

- **スワイプ遷移 (Touch Events)**: 
  - `touchstart`, `touchmove`, `touchend` イベントのリスナーを自作し、一定のピクセル以上X軸にスワイプされたら次/前の問題コンテナへ `transform: translateX()` 等のCSSアニメーションでなめらかに切り替える実装を行う。
- **TTS（Web Speech API）**:
  - 外部ライブラリ無しで対応可能な `SpeechSynthesis` APIを利用。（高度化する場合はGemini TTS APIのREST呼び出しに切り替え）。
  - 再生/停止、速度調整（1.0x, 1.2x, 1.5x）、再生中のハイライト表示などのUI制御をJS側で組む。
- **OCR（画像解析）のコンテキスト修復**:
  - プレビュー画面のエディタ（`textarea` や `input`）を通じて、画像解析後にパースされたJSONのテキスト部分を自由に書き換えられる。
  - 値が変更された時点で、内部のJSONステートを更新し、プレイ画面に流し込むデータフローをVanilla JSのイベントリスナーまたは簡易的なObserverパターンで管理する。
